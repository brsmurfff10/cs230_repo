1. Briefly summarize The Gaming Room client and their software requirements.

The client, The Gaming Room, wanted to expand their game Draw It or Lose It into a web-based, multiplayer application that could support multiple teams and players simultaneously. The software needed to be scalable, reliable, and accessible across different platforms, including desktop and mobile devices. Based on the design analysis, a Linux-based server environment was recommended due to its ability to handle large numbers of concurrent users, low cost, and strong support for cloud and distributed systems.

2. What did you do particularly well in developing this documentation?

I think I did well in clearly comparing different platforms (Linux, macOS, Windows, and mobile) and explaining how each one fits into the system. I focused on important factors like scalability, cost, and compatibility, which helped justify why Linux was the best choice for the server. I also made sure to connect technical decisions back to the client’s needs, rather than just listing features.

3. What about the process of working through a design document did you find helpful when developing the code?

Working through the design document helped me think about the system before writing any code. It forced me to consider things like architecture, platform choices, and user access early on. This made it easier to understand how different parts of the application would interact and reduced confusion later when actually developing the code.

4. If you could choose one part of your work on these documents to revise, what would you pick? How would you improve it?

If I could revise one part, I would improve the level of detail in the development considerations and team impact sections. I would include more specific examples, such as how deployment pipelines or cloud services would be set up. This would make the document more practical and easier to translate directly into implementation.

5. How did you interpret the user’s needs and implement them into your software design? Why is it so important to consider the user’s needs when designing?

I interpreted the user’s needs by focusing on scalability, accessibility, and ease of use. Since the game needed to support multiple players and teams, I designed it as a web-based application that could run across different devices using a browser. I also prioritized responsiveness for mobile users and reliability for the server. Considering user needs is important because the success of the application depends on how well it works for the people using it. If the design does not match user expectations, even technically correct software can fail.

6. How did you approach designing software? What techniques or strategies would you use in the future?

I approached the design by first identifying the requirements and constraints, then comparing different platform options based on those needs. I used a structured evaluation method to analyze trade-offs such as cost, scalability, and performance. In the future, I would continue using this approach but also incorporate more visual tools like diagrams and possibly prototype components earlier. I would also consider cloud-native design patterns and containerization sooner in the process to better align with modern development practices.
